# Triple-FP UVM-Lite

This folder adds a reusable plain-SystemVerilog verification scaffold around the four standalone triple-FP top modules:

- `TripleAddPipe_l4_f64`
- `TripleMulPipe_l4_f64`
- `TripleAddPipe_l4_f32`
- `TripleMulPipe_l4_f32`

It is intentionally "UVM-lite":

- no dependency on the full UVM library
- reusable request/response interfaces
- vector-driven stimulus acting as a sequence source
- scoreboard-style checking
- functional coverage for arithmetic scenario closure

## Files

- `triple_fp_uvm_lite_pkg.sv`
- `triple_fp_req_if.sv`
- `triple_fp_rsp_if.sv`
- `triple_fp_uvm_lite_cov.sv`
- `triple_fp_uvm_lite_env.sv`

- `tb_triple_fp_uvm_lite_f32.sv`
- `run_uvm_lite_verilator.sh`

## Architecture

The environment reuses the existing Python/TestFloat-generated vector files in:

- `/Users/kvsaiakhil/Projects/BoomV3/triple_fp_units/verif/vectors/vectors_f64_add.txt`
- `/Users/kvsaiakhil/Projects/BoomV3/triple_fp_units/verif/vectors/vectors_f64_mul.txt`
- `/Users/kvsaiakhil/Projects/BoomV3/triple_fp_units/verif/vectors/vectors_f32_add.txt`
- `/Users/kvsaiakhil/Projects/BoomV3/triple_fp_units/verif/vectors/vectors_f32_mul.txt`

Component split:

- driver:
  implemented inside `triple_fp_uvm_lite_env.sv` via `drive_case`
- sequence source:
  vector files generated by `generate_triple_fp_vectors.py`
- monitor:
  response interfaces plus env-side response sampling
- scoreboard:
  class-aware recFN result matching plus exception-flag comparison
- coverage:
  covergroup-based scenario coverage in `triple_fp_uvm_lite_cov.sv`

## Functional Coverage

Coverage includes:

- operation: add vs mul
- rounding mode
- operand classes for all three inputs
- output class
- special input combinations:
  zero / inf / NaN presence combinations
- sign patterns across the three operands
- invalid, overflow, underflow, and inexact flags

Crosses include:

- op x rounding mode
- op x special combination
- op x output class
- rounding mode x overflow
- rounding mode x underflow
- sign pattern x op

## Verilator Usage

The benches compile under Verilator. Covergroups are disabled automatically there via `VERILATOR` guards, so the flow still serves as a reusable scoreboard/driver harness even when coverage collection is unavailable.

### Quick run

```sh
/Users/kvsaiakhil/Projects/BoomV3/triple_fp_units/uvm_lite/run_uvm_lite_verilator.sh all
```

This supports `f64`, `f32`, or `all`.
