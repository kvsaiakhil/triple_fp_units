# Triple-FP UVM-Lite

This folder adds a reusable plain-SystemVerilog verification scaffold around the four standalone triple-FP top modules:

- `TripleAddPipe_l4_f64`
- `TripleMulPipe_l4_f64`
- `TripleAddPipe_l4_f32`
- `TripleMulPipe_l4_f32`

It is intentionally "UVM-lite":

- no dependency on the full UVM library
- reusable request/response interfaces
- vector-driven stimulus acting as a sequence source
- scoreboard-style checking
- functional coverage for arithmetic scenario closure

Assumed shell variables:

```sh
export BOOMV3_ROOT=/path/to/BoomV3
export REPO_ROOT="$BOOMV3_ROOT/triple_fp_units"
cd "$REPO_ROOT"
```

## Architecture

The environment reuses the existing Python/TestFloat-generated vector files in:

- `verif/vectors/vectors_f64_add.txt`
- `verif/vectors/vectors_f64_mul.txt`
- `verif/vectors/vectors_f32_add.txt`
- `verif/vectors/vectors_f32_mul.txt`

Component split:

- driver:
  implemented inside `triple_fp_uvm_lite_env.sv` via `drive_case`
- sequence source:
  vector files generated by `generate_triple_fp_vectors.py`
- monitor:
  response interfaces plus env-side response sampling
- scoreboard:
  class-aware recFN result matching plus exception-flag comparison
- coverage:
  covergroup-based scenario coverage in `triple_fp_uvm_lite_cov.sv`

## Functional Coverage

Coverage includes:

- operation: add vs mul
- rounding mode
- operand classes for all three inputs
- output class
- special input combinations:
  zero / inf / NaN presence combinations
- sign patterns across the three operands
- invalid, overflow, underflow, and inexact flags

Crosses include:

- op x rounding mode
- op x special combination
- op x output class
- rounding mode x overflow
- rounding mode x underflow
- sign pattern x op

## Verilator Usage

The benches compile under Verilator. Covergroups are disabled automatically there via `VERILATOR` guards, so the flow still serves as a reusable scoreboard/driver harness even when coverage collection is unavailable.

### Quick run

```sh
"$REPO_ROOT/uvm_lite/run_uvm_lite_verilator.sh" all
```

This supports `f64`, `f32`, or `all`.

### f64

```sh
verilator --binary --timing -Wall -Wno-fatal -Wno-UNUSEDSIGNAL \
  --top-module tb_triple_fp_uvm_lite_f64 \
  -Mdir "$REPO_ROOT/uvm_lite/obj_dir_f64" \
  "$REPO_ROOT/uvm_lite/tb_triple_fp_uvm_lite_f64.sv" \
  "$REPO_ROOT/uvm_lite/triple_fp_uvm_lite_pkg.sv" \
  "$REPO_ROOT/uvm_lite/triple_fp_req_if.sv" \
  "$REPO_ROOT/uvm_lite/triple_fp_rsp_if.sv" \
  "$REPO_ROOT/uvm_lite/triple_fp_uvm_lite_cov.sv" \
  "$REPO_ROOT/uvm_lite/triple_fp_uvm_lite_env.sv" \
  "$REPO_ROOT/TripleAddPipe_l4_f64.sv" \
  "$REPO_ROOT/TripleAddRecFNPipe_l2.sv" \
  "$REPO_ROOT/TripleAddRecFNToRaw.sv" \
  "$REPO_ROOT/TripleMulPipe_l4_f64.sv" \
  "$REPO_ROOT/TripleMulRecFNPipe_l2.sv" \
  "$REPO_ROOT/TripleMulRecFNToRaw.sv" \
  "$BOOMV3_ROOT/RoundRawFNToRecFN_e11_s53.sv" \
  "$BOOMV3_ROOT/RoundAnyRawFNToRecFN_ie11_is55_oe11_os53.sv"
"$REPO_ROOT/uvm_lite/obj_dir_f64/Vtb_triple_fp_uvm_lite_f64"
```

## Full-Simulator Usage

For Questa, VCS, or Xcelium, compile the same files without defining `VERILATOR` and the coverage collector will become active automatically.

## Finalization Role

This environment is intended to be the reusable verification landing zone for the project:

- the original directed benches remain for quick smoke tests
- the `verif/` benches remain the strong vector replay harness
- this `uvm_lite/` folder is the reusable structured verification scaffold for future growth
